

## 1. 目标与范围
- 设备端支持通过语音点歌并播放远端 URL 音乐（`.mp3、.wav`）。
## 2. 输入数据约定
- 歌单为多首列表。比如，用户说我想听歌（流行歌、儿歌等），默认可以推荐10首歌；然后用户自己可以随机说第一首或者具体歌名，就可以播放。
- 歌手点歌，比如我想听刘德华的歌，默认推荐10首歌，用户自己可以随机说第一首或者具体歌名，就可以播放。
- 具体歌名，比如我想听伍佰的“突然的自我”，直接播放具体的歌曲
- 每首歌至少包含：
  - `name`：歌曲名
  - `url`：可直接播放的 HTTP/HTTPS 地址
- 文件类型：`.mp3、.wav`
- 不需要额外授权流程。

## 3. 点歌与匹配规则
- 支持按歌曲名模糊匹配（如“我想听我只在乎你”）。
- 若模糊匹配到多首，默认播放第一首。
- 支持按序号播放（如“播放第二首”）。
- 若当前列表未命中指定歌曲，统一提示：
  - `当前列表没有这首，请说出正确歌曲名称。`

## 4. 播放队列与模式规则
- 产品有三个模式：对话模式、音乐模式和蓝牙模式（暂时没有蓝牙模式）
- 进入音乐模式后，维护当前播放清单（用于暂停/继续/上一首/下一首）。
- 暂停后继续，基于“上一轮/当前歌单”恢复，不丢失上下文。
- “继续播放”从**当前歌曲开头**重播（不是断点续播）。
- 歌单按顺序播放。
- 播放到最后一首后，**退出音乐模式**，不循环、不随机续播。

## 5. 语音控制指令
- 必须支持并生效：
  - `继续`
  - `下一首`
  - `上一首`
  - `暂停`
  - `恢复`
  - `退出音乐模式`
  - `进入对话模式`
- 指令在音乐播放过程中可直接打断并执行，不能“无响应”。

## 6. 唤醒与打断优先级（最高优先级）
- 播放音乐时必须可被唤醒词打断（唤醒词：`你好，小智`）。
- 唤醒打断触发后：
  - 退出音乐模式，进入对话模式。
  - 保留播放清单与当前歌曲索引，以便后续“继续”可回到对应歌曲。
- 要重点降低“播放声音干扰唤醒”的问题（提高打断成功率）。

## 7. UI 显示要求
- 音乐播放时，界面必须显示“正在播放”及当前歌曲名称。
- 状态切换（播放/暂停/退出/进入对话）需在界面同步反映，避免“像砖头”。

## 8. 稳定性与异常处理
- 处理“播放被无故打断”场景：不能造成模式卡死或控制失效。
- 任何退出/打断后，应保证可重新进入对话或重新点歌。

## 10. 验收建议（最小用例）
- 用例1：点歌（模糊命中单首）-> 正常播放 -> 显示歌曲名。
- 用例2：模糊命中多首 -> 仅播放第一首。
- 用例3：播放中说“下一首/上一首/暂停/继续”均生效。
- 用例4：播放中说唤醒词 -> 能打断进入对话模式。
- 用例5：说“退出音乐模式/进入对话模式”均可退出播放。
- 用例6：歌单最后一首播完 -> 自动退出音乐模式，不循环。
- 用例7：对话回复时 UI 与语音基本同步，无明显 1 秒级错位。
